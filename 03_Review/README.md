# 03 Review — Code vs Spec Conformity

> Conformity reports generated by `/review`. Objective GO/NO-GO score.

---

## When to run a review

- After building a component with `/build`
- When you want to verify an existing component conforms to its spec
- When you doubt the quality of an implementation

**Command**: `/review`

## How it works

1. **Prerequisite**: Code in `02_Build/{module}/` and a spec in `01_Product/05 Specs/{module}/specs/`
2. **Command**: `/review` compares code to spec
3. **Output**: A conformity report with X/Y score
4. **Verdict**: GO (100%) or NO-GO with list of typed gaps

## Template

The `_template-review.md` file shows the complete report format. Reports are generated automatically by `/review` — no need to fill manually.

## Reading a review report

Each review contains 5 verification levels:

| Section | Content |
|---------|---------|
| **Conformity score** | X/Y acceptance criteria passed (Gherkin) |
| **Complementary checks** | States (empty, loading, error, success), responsive, accessibility, types, tests |
| **UX checks** | UX laws (Miller, Hick, Fitts, Doherty, Jakob, Von Restorff, Goal-Gradient, Peak-End) |
| **Design System checks** | Hardcoded values, DS components, spacing, fonts |
| **Verdict** | **GO** (100%) or **NO-GO** with gap triage |

## NO-GO triage

If verdict is NO-GO, each gap is classified by type:

| Type | Means | Resolution agent |
|------|-------|------------------|
| **IMPL** | Bug or oversight in code | → `/build` |
| **SPEC** | Spec is incomplete or ambiguous | → `/spec` |
| **DESIGN** | UX pattern needs rework | → `/ux` |
| **DISCOVERY** | User hypothesis is wrong | → `/discovery` |

**Priority**: DISCOVERY > DESIGN > SPEC > IMPL
The dominant type determines which agent the review redirects to.

## This folder is empty?

Normal if you haven't run `/review` yet. You first need code (`/build`) to compare against a spec.
