# 02 Build — Source Code

> Code generated by `/build` from validated specs. Organized by module.

---

## Structure per module

```
02_Build/
└── {module}/              ← One folder per active module
    ├── src/               ← Source code
    │   ├── components/    ← React/Vue/etc. components
    │   ├── pages/         ← Pages/views
    │   ├── hooks/         ← Custom hooks
    │   ├── types/         ← TypeScript types
    │   ├── utils/         ← Utilities
    │   └── data/          ← Mock data, constants
    └── tests/             ← Tests (1 per acceptance criterion + 1 per state)
```

The `{module}` corresponds to the slug defined in `.claude/context.md` and `modules-registry.md`.

## How it works

1. **Prerequisite**: A validated spec (status `VALIDATED`) in `01_Product/05 Specs/{module}/specs/`
2. **Command**: `/build` reads the spec and generates tests then code (TDD)
3. **Output**: Files in `02_Build/{module}/src/` and `02_Build/{module}/tests/`

## Rules

- **No code without spec** — `/build` refuses if spec isn't validated
- **Tests first** — Tests are written before code
- **4 mandatory states** — Each component handles: empty, loading, error, success
- **Design System** — Zero hardcoded values, always use tokens from `05 Design System/`
- **Zero dead-ends** — Every button has a handler, every link leads somewhere

## This folder is empty?

Normal if you haven't run `/build` yet. The recommended cycle:
1. `/ux` → explore directions
2. `/spec` → generate spec
3. `/build` → code from spec (files appear here)
4. `/review` → verify conformity

## Iteration log

Each build session creates a log in `02_Build/iterations/iteration-XXX.md` to trace decisions made.
