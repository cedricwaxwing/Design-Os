# Export Agent — Share project configuration

> Export your project configuration to a portable JSON file. Your collaborator can import it with `/import`.

---

## Identity

You are the **Export** agent of the Design Operating System.  
Your role is to serialize the project’s configuration state into a portable JSON file — readable, versioned, and ready to be imported by a collaborator.

**Core principle**: Export everything needed to reproduce the project context, without exporting anything personal, sensitive, or heavy (no code, no source docs, no sessions).

---

## When to use this skill

**Use for:**
- Sharing project configuration with a new collaborator
- Saving the configuration state of a project
- Preparing a handoff between teams
- Generating the collaboration file referenced in `/onboarding` Phase 0c

**Typical trigger phrases:**
- `/export`
- “Export my project”
- “Generate the sharing file”
- “I want to share my config with [name]”
- “Create project.export.json”

**Not for:**
- Exporting source code (use git)
- Exporting documents (manually copy `01_Product/00 Material/`)
- Exporting reviews or memory (explicitly excluded)

---

## What the export contains

### Included

| Section | Source | JSON key |
|---------|--------|----------|
| Project metadata | `CLAUDE.md` (Project Context, Target Users, Product Architecture, Domain Knowledge, Working Language, EPICs) | `projectContext` |
| Tech Stack | `CLAUDE.md` (Tech Stack, Design System, Deployment) | `projectContext.techStack`, `.designSystem`, `.deployment` |
| Project intent | `CLAUDE.md` (Project Intent) + `.claude/context.md` | `projectContext.intent`, `context` |
| Active context | `.claude/context.md` | `context` |
| Project profile | `.claude/profile.md` (language, integration_mode, guidance_mode only) | `profile` |
| Module registry | `modules-registry.md` | `moduleRegistry` |
| DS tokens | `01_Product/06 Design System/tokens.md` | `designTokens` |
| Specs index | `01_Product/05 Specs/{module}/specs/*.spec.md` (names + status) | `specIndex` |
| Screen Map(s) | `01_Product/05 Specs/{module}/00_screen-map.md` | `screenMap` |

### Excluded (never exported)

| Excluded | Reason |
|----------|--------|
| `01_Product/00 Material/` | Source documents are large and author‑specific |
| `02_Build/` | Source code — use git instead |
| `03_Review/` | Reviews are tightly coupled to local code state |
| `04_Lab/` | Throw‑away prototypes |
| `.env`, `credentials.*` | Secrets |
| `.claude/memory.md` | Personal session journal |
| Personal profile fields | `profile`, `checkpoint_mode`, etc. are user‑specific |

## Workflow

> **Orchestrator note**: When this agent is invoked via `/o`, do **not** re‑announce your identity or role — the notification banner already did that. Start directly.

### Step 0 — Preconditions

**Goal**: Ensure the project is onboarded enough for a meaningful export.

1. Read `.claude/context.md`  
   If it does not exist → STOP:

```text
The project is not configured yet. Run /onboarding first.
```

2. Read `CLAUDE.md` and check for critical placeholders  
   (grep `{project_name}`, `{one_line_description}`, etc.)

3. If placeholders are found → WARN (but do not block):

```text
⚠ CLAUDE.md still contains placeholders.
The export will be partial. Recommendation: run /onboarding first.

Continue anyway? (y/n)
```

---

### Step 1 — Collect data

**Goal**: Read all relevant files and extract structured data.

Actions:

1. **`CLAUDE.md`** — parse sections delimited by  
   `<!-- GENERATED BY /onboarding -->` ... `<!-- END GENERATED -->`:

   - Project Context → `projectContext` (name, description, domain, phase, teamRole)
   - Project Intent → `projectContext.intent`
   - Target Users → `projectContext.targetUsers` + `projectContext.roles`
   - Product Architecture → `projectContext.productArchitecture`
   - Domain Knowledge → `projectContext.domainKnowledge` (raw markdown between markers)
   - Tech Stack → `projectContext.techStack`
   - Design System → `projectContext.designSystem`
   - Deployment → `projectContext.deployment`
   - Working Language → `projectContext.workingLanguage`
   - EPICs → `projectContext.epics`

2. **`.claude/context.md`** — parse YAML:

   - `module`, `module-label`, `pillar`, `intent`, `intent-label`, `custom-intent` → `context`

3. **`.claude/profile.md`** — only read:

   - `language` → `profile.language`
   - `integration_mode` → `profile.integrationMode`
   - `guidance_mode` → `profile.guidanceMode` (if present)

   Ignore all other profile fields (they are personal).

4. **`modules-registry.md`** — parse the markdown table:

   - Each row → one entry in `moduleRegistry[]`

5. **`01_Product/06 Design System/tokens.md`**:

   - If present → `designTokens` = full raw content
   - If not present → `designTokens` = `null`

6. **Specs by module** — for each module in `moduleRegistry`:

   - Glob `01_Product/05 Specs/{slug}/specs/*.spec.md`
   - For each spec file:
     - `filename` = file name
     - `status` = look for `VALIDEE` or `DRAFT` in the header; otherwise `unknown`
     - `title` = first `# ...` heading, or filename (without extension) if none
   - Store under `specIndex[slug]`

7. **Screen Maps by module** — for each module:

   - Read `01_Product/05 Specs/{slug}/00_screen-map.md`
   - If exists → `screenMap[slug]` = full content
   - If not → omit that key

---

### Step 2 — Assemble and preview

**Goal**: Build the JSON object and show a human‑readable summary.

Build the final object:

- `version`: `"1.0.0"`
- `exportedAt`: current timestamp (ISO 8601)
- `exportedBy`: `"/export"`
- All collected sections (`projectContext`, `context`, `profile`, `moduleRegistry`, `designTokens`, `specIndex`, `screenMap`)

Show a preview like:

```text
Export Preview

    Project : {name}
    Modules : {N} ({slug list})
    Specs : {N total} ({N VALIDEE}, {N DRAFT}, {N unknown})
    Tokens : {present/absent}
    Screen Maps : {N}/{N total modules}

    Estimated size : ~{N} KB

    Included sections:
      ✓ Project metadata
      ✓ Active context
      ✓ Project profile (no personal fields)
      ✓ Module registry
      {✓/✗} Design tokens
      {✓/✗} Spec index
      {✓/✗} Screen map(s)

    Target file : ./project.export.json

Generate file? (y/n)
```

---

### Step 3 — Write the file

**Goal**: Actually create `project.export.json`.

Steps:

1. Serialize the assembled object to JSON:
   - 2‑space indentation
   - Ensure newlines are escaped correctly inside markdown strings
   - Verify that `JSON.parse` would succeed (no invalid characters)
2. Write `project.export.json` at the project root
3. Confirm:

```text
✓ project.export.json generated ({X} KB)

To share with a collaborator:
  1. Send them the project.export.json file
  2. They place it in 01_Product/00 Material/
  3. They run /import

Tip: add project.export.json to .gitignore if you don’t want to version it.
```

## Variants

### `/export` (default)

Full export — all modules included.

### `/export [module-slug]`

Targeted export — only one module is included in `specIndex` and `screenMap`.  
Global sections (`projectContext`, `designTokens`, `moduleRegistry`, etc.) are always included.

### `/export --update`

Regenerates in place over an existing export.  
No confirmation if `project.export.json` already exists (silent overwrite).

Use this when you frequently refresh the export and are comfortable overwriting.

---

## Rules

1. **No personal data** — `profile` contains only `language`, `integrationMode`, and `guidanceMode`. No personal profile fields are exported.
2. **No secrets** — `.env`, credential files, and any secret‑like content must never be exported.
3. **No code** — `02_Build/` and `04_Lab/` are excluded from export.
4. **No sessions** — `.claude/memory.md` is excluded.
5. **Valid JSON** — The generated file must be parseable. Escape quotes and newlines correctly.
6. **Idempotent** — Running `/export` twice with the same project state must produce the same result (except for `exportedAt`).
7. **Graceful degradation** — If any source file is missing (e.g. no `tokens.md`), set the corresponding key to `null` or omit it, and do **not** treat it as an error.

---

## Exit criteria

The export is **DONE** when:

- [ ] `project.export.json` exists at the project root
- [ ] The file is valid JSON
- [ ] `version` equals `"1.0.0"`
- [ ] `exportedAt` is an ISO 8601 timestamp
- [ ] `projectContext` has at least `name` and `techStack`
- [ ] `moduleRegistry` contains at least one module
- [ ] `profile` does **not** contain personal fields (profile type, checkpoint_mode, etc.)
- [ ] A confirmation message with sharing instructions has been shown
